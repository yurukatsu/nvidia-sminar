## 全体まとめ

楽天グループが実施した **金融特化 Vertical LLM** の研究開発プロジェクトの報告。「金融向け AI は"使う"から"所有して育てる"へ」をキーメッセージに、Qwen 30B を基盤モデルとして SFT（教師ありファインチューニング）を実施し、日本株式の騰落予測タスクで評価した。

NVIDIA NeMo Curator をデータキュレーションに活用し、金融一般知識・企業財務情報・市場動向データを統合。SFT により基盤モデル比 **+4%〜+10.55%** の精度改善を確認し、自社タスクでは **GPT-5 を含む参照モデルよりも高い精度**を達成。一方で、金融データの統合・入力設計の複雑性が高く、**データ品質と金融ドメイン知識がモデル性能を大きく左右する**ことが主要な学びとなった。

## Key Points

### 1. プロジェクトの背景と目的

- **目的**：金融タスクに特化した LLM の実現可能性を検証し、汎用モデルとの差異・実務適応性に向けた論点を明らかにする
- **動機**：
  - 金融特有の機密性・安全性要件により汎用 LLM の利用に限界がある
  - 金融業務特有の文脈・専門用語・判断基準に汎用モデルが追いつけないケースがある
  - ファインチューニングの実務上の負荷と性能改善の程度を定量的に把握したい
- **体制**：楽天 AI & データ本部の2チーム
  - **AI & データコンサルティング部**：ビジネス要件・プロジェクト管理・データ調達・品質管理・ベンチマーク設計
  - **楽天技術研究所（RIT）**：リサーチ・モデル開発・エンジニアリング（世界6拠点：東京、ボストン、シンガポール、サンマテオ、バンガロール、パリ）

### 2. 学習アプローチと評価タスク

**学習手法**：SFT（Supervised Fine-Tuning）を選択。効果と課題の可視化が目的のため、制御性が高く比較検証に適した手法として選定。

**評価タスク** — 日本株式の騰落予測（3タスク）：

| タスク | 内容 | 分類 | 対象 |
| --- | --- | --- | --- |
| **目標株価** | 財務情報から証券アナリストの目標株価の騰落を予測 | 上昇/維持/下落（3クラス） | 大型株中心 300社 |
| **株価** | 決算発表後の株価騰落を予測（T+1 / T+7） | 上昇/下落（2クラス） | 大型株中心 300社 |
| **指数（ETF）** | マクロ経済指標発表後の業種別指数 ETF の騰落を予測 | 上昇/下落（2クラス） | TOPIX 17シリーズ |

**選定理由**：正解ラベルが明確でデータ収集が容易。他の候補（IR コンテンツ作成、エンゲージメント活動支援、個人資産分析、VC のビジネスデューデリジェンス等）の中から選定。

### 3. データ設計と NeMo Curator の活用

**データソース（3カテゴリ）**：

| カテゴリ | ソース |
| --- | --- |
| 金融一般知識 | Web 上の金融業界用語、HuggingFace 上の金融知識 Q&A・金融数理推論 Q&A |
| 企業財務情報 | EDINET、金融データ提供企業（財務情報、株価実績、目標株価） |
| 金融市場動向 | Web 上のマクロ経済指標・ニュース、金融データ提供企業の市場センチメント |

**データ量**：

| データセット | 学習 | 評価 |
| --- | --- | --- |
| 目標株価 | 約 170,000 件 | 約 27,000 件 |
| 株価 | 約 170,000 件 | 約 8,800 件 |
| 指数 | 約 200,000 件 | 約 7,000 件 |
| 金融用語定義・財務情報 Q&A | 約 120,000 件 | 約 8,200 件 |

**SFT データの構造例**（株価騰落予測）：

- 入力：「あなたは日本株分析専門の金融アナリストであり、以下は〇〇社の2025年第1四半期決算内容及び一般的なテクニカル指標です。この決算が公表された後の翌営業日の終値が上昇か下落のみを出力してください」
- 出力（ラベル）：「アップ（上昇）」

**NeMo Curator の活用**：

多様なデータソースから収集したテキストに対し：
- **重複除去**：データの冗長性を排除
- **品質フィルタリング**：品質スコアによるフィルタ
- **形式統一**：フォーマットの標準化
- SFT 学習に適した高品質データへ整形

NeMo Curator の強み：
- **導入容易性**：OSS かつ Python ライブラリ
- **環境適応性**：CPU〜GPU クラスターまでスケール可能
- **データ処理機能**：テキスト・画像・音声・動画に対応

### 4. モデル開発

- **基盤モデル**：Qwen 30B（オープンソース、高性能、日本語対応、継続的改善、多くの企業で導入実績）
- **学習手法**：SFT（クロスエントロピー損失の最小化）
- **比較対象**：基盤モデル（Qwen 30B）、ファインチューニングモデル（Fin-LM）、GPT-5（ベースライン）

### 5. 評価結果

**自社ベンチマーク**：

| タスク | 基盤モデル比改善 | vs GPT-5 |
| --- | --- | --- |
| 目標株価 | **+5.45 pt** | GPT-5 を上回る |
| 株価 | **+10.19 pt** | GPT-5 を上回る |
| 指数 | **+4 pt** 程度 | GPT-5 を上回る（一部指標を除く） |

→ 全体として基盤モデル比 **最大 +10.55 pt**、自社タスクでは GPT-5 を含む全参照モデルよりも高精度

**オープンソースベンチマーク**：

| ベンチマーク | タスク | 結果 |
| --- | --- | --- |
| PFN 日本語 LM Financial Evaluation Harness | CHABSA（センチメント分析） | 基盤モデルを上回る |
| 同上 | CMA Basics（金融基礎知識） | 基盤モデルを上回る |
| 同上 | Security Sales 1（外務員試験模試） | **基盤モデルを下回る**（要因分析中） |
| Sakana AI EDINET Bench | Earning Forecast（業績予想） | 基盤モデル比 +3.3 pt だが GPT-5 には及ばず |

EDINET Bench で GPT-5 に及ばなかった点については、「元データが GPT-5 の学習コーパスに含まれていた可能性がある」と推測。

### 6. 学びと実務導入への提言

**主要な学び — 「データが品質を決める」**：

| フェーズ | 学び |
| --- | --- |
| **データ選定** | LLM 知識だけでなく金融ドメインの専門知識が不可欠。データ形式の設計にも影響 |
| **アノテーション** | 非常に工数がかかる作業。必要人材の明確化が重要 |
| **データ収集** | 金融データの LLM ファインチューニングへの利用には法務確認が必要。収集工程自体がコスト要因 |
| **SFT データ作成** | データの多様性確保と複数ソースの統合精度が精度を左右 |
| **評価** | 初期はデータ量・分布の影響による偏りが発生。適切な評価基盤と定期的な再評価プロセスが必要 |

**実務導入に必要な3つの要素**：

1. **LLM 知識 × ドメイン知識によるデータ統合**：財務・株価・指標・ニュース等の異種データの統合には、定義・期間・粒度の違いを理解した統合ロードマップが必要
2. **スケーラブルなユースケース選定**：再現性が高く、評価しやすく、ビジネス価値が高いタスクから着手。SFT に加え RAG やプロンプトエンジニアリングの組み合わせも重要
3. **運用基盤（MLOps / LLMOps）の整備**：モデル更新頻度、学習バージョン管理、算定プロセス、データドリフト検知 → 「自社データでモデルを育て続ける」ための基盤

## 当チームへの示唆

- **Vertical LLM の実現可能性が実証済み**：Qwen 30B + SFT で GPT-5 を上回る金融タスク精度を達成した事例。当チームでも市場データ分析・投資戦略開発の特定タスクで同様のアプローチが検討可能
- **NeMo Curator の有用性**：金融データの前処理・キュレーションに NeMo Curator が有効であることが確認された。当チームが大量の金融テキストデータを扱う際のツール候補として検討に値する
- **データ品質 > モデルアーキテクチャ**：金融ドメイン知識に基づくデータ選定・統合・アノテーションがモデル性能の決定要因。LLM 活用プロジェクトでは、モデル選定よりもデータパイプラインの設計に注力すべき
- **法務確認の重要性**：金融データを LLM のファインチューニングに使用する際の法務確認プロセスは、当チームでも同様に必要となる論点
- **ユースケース選定の方法論**：「正解ラベルが明確」「データ取得が容易」「ビジネス価値が高い」の3軸で評価するアプローチは、当チームの生成 AI 活用プロジェクトの優先順位付けに適用できる
- **MLOps / LLMOps の必要性**：モデルを「育て続ける」ための運用基盤（更新頻度、バージョン管理、データドリフト検知）は、当チームが AI エージェント活用を拡大する際にも不可欠な基盤
